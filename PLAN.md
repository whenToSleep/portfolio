# План доработки сайта-портфолио с помощью Claude Code

**Дата:** 15 июня 2026
**Текущее состояние:** прототип `portfolio.html` — одностраничное React-приложение, скомпилированное в браузере через Babel-standalone. Весь контент захардкожен в JS-объектах (`ARTIST`, `WORKS`, `PROJECT`, `LETTER`, `STR`), иллюстрации — серые плейсхолдеры (`.plate`). Двуязычность EN/UK реализована вручную через объект `STR` и функцию `L()`.

---

## 1. Целевой стек (решения приняты)

| Слой | Выбор | Почему |
|---|---|---|
| Фреймворк | **Next.js (App Router)** | Прототип уже на React — компоненты переносятся почти 1:1. SSR/ISR, оптимизация изображений, готовая интеграция с CMS. |
| CMS / админка | **Payload CMS 3** | Единственная CMS, которая ставится **прямо внутрь Next.js-приложения** (один репозиторий, один деплой). Готовая простая админка, медиабиблиотека, встроенная локализация RU/UK/EN, типобезопасность (TypeScript). Бесплатна и self-hostable. |
| БД | **Postgres** (Neon на стейдже → свой Postgres на VPS) | Payload поддерживает Postgres из коробки; Neon даёт бесплатный managed-инстанс для стейджа. |
| Хранилище медиа | **Vercel Blob** на стейдже → **S3-совместимое** (Cloudflare R2 / MinIO) на VPS | Адаптеры storage в Payload меняются одной строкой конфига. |
| Хостинг | **Vercel** (стейдж/разработка) → **VPS + Docker** (прод) | Vercel даёт мгновенный preview-деплой каждой ветки. Payload официально разворачивается и serverless на Vercel, и в контейнере на VPS. |

**Почему не отдельная headless CMS (Sanity/Strapi) и не git-CMS (Tina/Keystatic):** Payload даёт ту же мощь, что Sanity/Strapi, но живёт в том же Next.js-проекте и том же деплое — меньше движущихся частей. В отличие от git-CMS, у него полноценная медиабиблиотека и удобная локализация полей — а это ровно требование «менять все тексты и все изображения».

## 2. Целевая архитектура

```
portfolio/                      # один Next.js + Payload репозиторий
├─ app/
│  ├─ (frontend)/               # публичный сайт
│  │  ├─ [lang]/                # /en, /uk (и /ru при желании)
│  │  │  ├─ page.tsx            # Cover (Home)
│  │  │  ├─ index/page.tsx      # The Index (works)
│  │  │  ├─ work/[slug]/page.tsx# Project
│  │  │  ├─ letter/page.tsx
│  │  │  └─ masthead/page.tsx
│  │  └─ layout.tsx             # topbar, тема, шрифты, курсор
│  └─ (payload)/admin/          # админка Payload (генерируется)
├─ src/
│  ├─ collections/              # Works, Media, Pages, Users
│  ├─ globals/                  # SiteSettings, Masthead, Letter, Home
│  ├─ components/               # перенесённые из прототипа React-компоненты
│  └─ payload.config.ts
└─ ...
```

Идея: **всё, что сейчас в `STR`/`WORKS`/`ARTIST`/`PROJECT`/`LETTER`, становится контентом в Payload**, локализованным по языкам. Фронтенд читает данные через локальный Payload API на сервере и рендерит те же визуальные компоненты.

## 3. Контент-модель Payload (что на что мапится)

| Сущность в прототипе | Тип в Payload | Поля (локализованные отмечены 🌐) |
|---|---|---|
| `WORKS[]` (12 работ) | **Коллекция `works`** | `num`, `title`🌐, `client`, `year`, `tags[]`, `coverImage`→Media, `gallery[]`→Media, `body`🌐 (rich text), `slug` |
| `TAGS` / `TAG_ORDER` | **Коллекция `tags`** или select-поле | `label`🌐, `order` |
| `ARTIST` | **Global `siteSettings`** | `name`🌐, `city`🌐, `email`, `commissions`, `press`, соцсети[] |
| `PROJECT` (шаблон статьи) | поля внутри `works` | переносится в `body`/`subtitle`/`pull` каждой работы |
| `LETTER` | **Global `letter`** | `dek`🌐, `paragraphs`🌐 (rich text), `signature`🌐 |
| `STR` (строки Home/Index/Masthead) | **Globals** `home`, `masthead` + словарь UI-строк | все подписи, метки, статусы — локализованные поля |
| `.plate` плейсхолдеры | поля типа **Media (upload)** | реальные изображения с автогенерацией размеров |

**Локализация:** в `payload.config.ts` включаем `localization: { locales: ['en','uk','ru'], defaultLocale: 'en' }`. Тогда каждое поле с `localized: true` хранит версию на каждом языке, а админка показывает переключатель языка — пользователь правит EN и UK (и RU) в одном месте.

## 4. Итерационный план (фазы для Claude Code)

Каждая фаза — самостоятельный, проверяемый кусок работы. Отдавай Claude Code по одной фазе, в конце каждой — проверка (билд + визуальное сравнение со скриншотом прототипа). Коммить после каждой фазы.

### Фаза 0 — Подготовка и фиксация эталона
**Цель:** зафиксировать, как должен выглядеть результат, до любых изменений.
- Снять скриншоты всех 5 экранов прототипа в обоих языках и обеих темах (день/ночь) — это визуальный эталон для регрессии.
- Завести git-репозиторий, ветку `main` + `dev`.
- Вынести из `portfolio.html` в отдельные заметки: палитру (`:root` переменные), список шрифтов Google Fonts, все тексты `STR` (понадобятся как сид контента).
> **Готово, когда:** есть репо, скриншоты-эталоны и выписанные токены/тексты.

### Фаза 1 — Каркас Next.js + перенос статики «как есть»
**Цель:** тот же сайт, но уже на настоящем Next.js (без Babel-в-браузере), контент пока захардкожен.
- `create-next-app` (TypeScript, App Router).
- Перенести CSS из `<style>` в глобальный CSS / CSS-модули; подключить шрифты через `next/font`.
- Разбить монолитный React на компоненты: `Topbar`, `HomePage`, `IssuePage`, `ProjectPage`, `LetterPage`, `MastheadPage`, `Plate`, хук `useReveal`, кастомный курсор.
- Роутинг через App Router вместо самописного `go()`; сохранить view-transition анимации.
- Данные временно положить в `lib/content.ts` (те же объекты `WORKS`/`STR`).
> **Готово, когда:** `next build` проходит, все 5 страниц визуально совпадают с эталоном из Фазы 0, переключение языка и темы работает.
> **Подсказка для промпта:** «Перенеси этот компонент 1:1, не меняя вёрстку и анимации; цель — пиксель-в-пиксель как в прототипе».

### Фаза 2 — Локализация на уровне маршрутов
**Цель:** заменить самописный объект `STR`/`L()` на стандартный i18n-роутинг.
- Префиксы языка в URL: `/en`, `/uk`, `/ru` через сегмент `[lang]`.
- Middleware для дефолтного языка и переключателя.
- Тексты пока остаются в словарях, но читаются по активному locale.
> **Готово, когда:** `/uk/...` и `/en/...` отдают правильный язык на сервере (SSR), переключатель в topbar меняет URL.

### Фаза 3 — Установка Payload и контент-модель
**Цель:** появляется админка и схема данных (пока пустая).
- Поставить Payload 3 в существующий Next.js (`npx create-payload-app` поверх / официальный one-line install).
- Поднять Neon Postgres (стейдж), подключить Vercel Blob для медиа.
- Описать коллекции `works`, `tags`, `media`, `users` и глобалы `siteSettings`, `home`, `letter`, `masthead` (по таблице из раздела 3).
- Включить `localization` с locales en/uk/ru.
- Настроить доступ: публичное чтение, запись — только авторизованным.
> **Готово, когда:** `/admin` открывается, можно залогиниться, создать тестовую работу с картинкой, переключить язык поля.

### Фаза 4 — Сид контента и подключение фронтенда к Payload
**Цель:** сайт рендерится из CMS, а не из захардкоженных объектов.
- Написать seed-скрипт: перенести все 12 работ, тексты `STR`, `ARTIST`, `LETTER` в Payload (EN + UK; RU добавить позже).
- Заменить импорты из `lib/content.ts` на запросы к Payload Local API в серверных компонентах.
- Плейсхолдеры `.plate` заменить на `next/image` от загруженных Media (с фолбэком на плейсхолдер, если изображение не задано — важно, т.к. картинки «будут позже»).
> **Готово, когда:** правка текста/картинки в `/admin` меняет публичный сайт; пустые изображения деградируют в плейсхолдер без ошибок.

### Фаза 5 — Упрощение админки под пользователя
**Цель:** «простая для восприятия» админка (явное требование).
- Сгруппировать поля по вкладкам, добавить русские/понятные подписи и подсказки (`admin.description`).
- Настроить порядок и группы в сайдбаре; спрятать технические поля.
- Преднастроить превью изображений и осмысленные имена коллекций («Работы», «Главная», «Лист», «Контакты»).
- Включить Live Preview / Draft+Publish, чтобы пользователь видел изменения до публикации.
> **Готово, когда:** человек без опыта может сам поменять любой текст и любую картинку и опубликовать, не путаясь.

### Фаза 6 — Деплой на Vercel (стейдж)
**Цель:** рабочий публичный стейдж.
- Подключить репо к Vercel, env-переменные (DB, Blob, секрет Payload).
- Проверить, что админка и сайт работают в serverless-окружении.
- Настроить ISR/ревалидацию: публикация в CMS обновляет страницы.
> **Готово, когда:** стейдж-URL открывается, правки в админке появляются на сайте.

### Фаза 7 — Изображения, SEO, доступность, производительность
**Цель:** довести до прод-качества.
- Размеры изображений (`imageSizes`), `next/image`, lazy-load, blur-placeholder.
- SEO: метатеги, OpenGraph, `sitemap.xml`, `robots.txt`, hreflang для языков.
- Доступность (можно прогнать скиллом accessibility-review): контраст, фокус, prefers-reduced-motion (уже частично есть), alt-тексты из CMS.
- Lighthouse / Core Web Vitals.
> **Готово, когда:** Lighthouse ≥ 90 по всем осям, hreflang и OG корректны.

### Фаза 8 — Переезд на VPS (прод)
**Цель:** self-hosted прод, как и планировалось.
- Dockerfile + docker-compose: Next.js+Payload, Postgres, обратный прокси (Caddy/Nginx) с TLS.
- Заменить Vercel Blob на S3-совместимое хранилище (Cloudflare R2 или MinIO).
- Бэкапы БД и медиа; CI на пуш в `main`.
- Перенос контента со стейджа (дамп Postgres + медиа).
> **Готово, когда:** прод-домен на VPS работает по HTTPS, бэкапы настроены, деплой автоматизирован.

## 5. Технические нюансы и риски (на что заранее обратить внимание Claude Code)

- **Babel-standalone уходит полностью.** В проде компиляция в браузере недопустима — весь JSX собирается Next.js. Это основная причина миграции.
- **Кастомные анимации.** View Transitions, scroll-reveal (`useReveal`), кастомный курсор и тема день/ночь — перенести аккуратно; это «характер» сайта. Проверять визуально после каждой фазы.
- **`dangerouslySetInnerHTML` для строк с `<em>`.** Сейчас тексты содержат HTML-разметку. В CMS лучше перейти на rich text (Lexical в Payload) или явный sanitization — не тащить сырой HTML из пользовательского ввода без очистки.
- **Шрифты.** 6 семейств Google Fonts — через `next/font` для производительности и стабильного рендера.
- **Картинки «будут позже».** Обязателен graceful-фолбэк на `.plate`-плейсхолдер, иначе пустые работы сломают вёрстку.
- **Tweaks-панель** из прототипа — это инструмент дизайна, в прод не переносится (удалить весь блок `tweaks-panel.jsx`).
- **Три языка.** Сейчас EN/UK; ты пишешь на RU — заложи RU как третий locale сразу в Фазе 3, даже если контент добавишь позже.

## 6. Как работать с Claude Code по этому плану

1. Открой папку проекта в Claude Code, дай ему прочитать `portfolio.html` и этот `PLAN.md`.
2. Запускай **по одной фазе за сессию** — проси план реализации фазы, ревьюй, затем выполнение.
3. После каждой фазы: `next build`, сравнение со скриншотами-эталонами, коммит.
4. Держи отдельную ветку на фазу, мерж в `dev` после проверки.
5. Для рискованных мест (анимации, локализация) проси Claude Code сначала показать diff, потом применять.

**Порядок зависимостей:** 0 → 1 → 2 → 3 → 4 → 5 → 6, далее 7 и 8 (7 можно частично параллелить с 6).
